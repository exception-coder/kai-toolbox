# req-check — 需求评审技能

当用户通过 `/req-check` 触发此技能时，对紧跟其后的需求进行系统性评审，**在动手实现之前**完成分析。

## 评审四维

### ① 可行性
- 当前架构/技术栈（Java 21 / Spring Boot 3.4 / React 19）是否原生支持？
- 是否有隐性依赖、需要引入新中间件？
- 实现难度与工作量粗估

### ② 价值
- 解决什么具体问题？受益对象是谁？
- 是否与现有功能重复或冲突（参考 CLAUDE.md 的模块列表）？
- 与项目整体方向（单用户本地工具平台）是否吻合？

### ③ 风险
- **安全**：是否暴露敏感数据或接口？
- **性能**：高频操作？大数据量？阻塞主线程？
- **破坏性**：是否影响现有功能/接口兼容性？
- **技术债**：是否绕过架构约定（如 §6 明确排除项）？

### ④ 结论

用以下固定格式给出结论，不要跳过：

```
✅ 建议实施
  理由：...
  （无特殊条件则直接进入实施）

⚠️ 有条件实施
  前提条件：...
  调整建议：...

❌ 建议拒绝
  原因：...（必须具体，不能模糊）
  替代方案：...（如有）
```

## 使用规则

- **✅ 是默认倾向**：大多数需求应给出"建议实施"或"有条件实施"，不要轻易拒绝
- **❌ 仅用于**：明显无意义的重复需求、违反 §6 架构约束的需求、或安全/性能风险极高无法缓解的需求
- **评审完成后**：如结论是 ✅ 或 ⚠️ 且用户确认，立即开始实施；如结论是 ❌，等待用户反馈
- **简洁优先**：评审输出用要点列表，不需要写成长篇文章

## 触发方式

```
/req-check 帮我做一个 XXX 功能
```

或：在对话中先说明需求，然后单独发 `/req-check` 触发对上文最近需求的评审。
